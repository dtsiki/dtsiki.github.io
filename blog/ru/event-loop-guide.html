Синхронность и асинхронность

— это про то, как в языке выполняются операции в коде:

  • синхронно:

    1. Операции выполняются последовательно и строго по очереди
    2. Каждая новая операция ждет завершения предыдущей
    3. Если одна из операций зависнет или потребует больше времени, вся программа зависнет и будет ждать её выполнения

    Пример из жизни: вы захотели выпить кофе. Идёте на кухню, включаете чайник и стоите перед ним, не двигаясь, пока вода не закипит. Ничего другого вы не делаете — просто ждёте. Проходит 3 минуты. Чайник закипает. Теперь вы идёте к шкафу, берёте кружку и обнаруживаете, что она грязная. Приходится мыть посуду. Затем достаёте молоко, греете его в капучинаторе, завариваете кофе и наконец пьёте. Весь процесс занимает ±10 минут. Из них целых 5 минут вы просто стоите, ничего не делаете и смотрите на чайник.

    В коде это можно оформить так:

    snippet(1).js
    boilWater();      // 5 минут стоим и ждём чайник, программа дальше не идёт
washDishes();     // 1 минута
takeCup();        // 30 секунд
getMilk();        // 30 секунд
steamMilk();      // 1 минута
getCoffee();      // 1 минута

  • асинхронно:

    1. Выполняется не в строгой последовательности
    2. Результат выполнения может быть доступен не сразу, а через некоторое время
    3. Если одна из операций зависнет или потребует больше времени, вся программа не зависнет и не будет ждать её выполнения, а просто пойдёт дальше

    Пример из жизни: вернёмся к кофе из примера выше. Идёте на кухню, включаете чайник и не ждёте. Пока вода греется, вы идёте мыть кружку. Помыли. Достаёте молоко из холодильника и ставите его рядом. Чайник всё ещё греется. Вы успеваете даже подготовить капучинатор. Наконец, чайник закипает — вы слышите свист (это как «callback» от чайника). Вы завариваете кофе, греете молоко и пьёте. Всё то же самое количество дел, но вы ни секунды не стояли без дела. Чайник кипятился параллельно с вашими действиями.

    В коде это можно оформить так:

  // Этот колбэк выполнится только когда чайник закипит
  getCoffee();
});

getMilk();

Однопоточность и многопоточность

— это про то, сколько дел может выполняться физически одновременно:

  • однопоточность:

    1. Вся программа имеет только один поток выполнения, поэтому в один момент времени может выполняется только что-то одно
    2. Чтобы сделать несколько дел одновременно, нужно использовать асинхронность: начать операцию, а когда она завершится — получить уведомление. Но в сам момент выполнения всё равно занят только один поток.

    Пример из жизни: один повар в ресторане. Приходит заказ: приготовить три блюда. Он готовит первое блюдо от начала до конца, потом второе, потом третье. Клиенты ждут, пока повар приготовит каждое блюдом по очереди, потому что повар не может одновременно жарить картошку и нарезать салат — руки-то одни.

  • многопоточность:

    1. Может быть несколько потоков, которые выполняются параллельно — на разных ядрах процессора или в вытесняющем режиме
    2. Эти несколько потоков могут работать одновременно и не ждать друг друга

    Пример из жизни: вернёмся к повару из примера выше, но тут у него будут помощник повара и стажёр. Пока повар готовит одно блюдо, помощник повара занимается вторым, а стажёр третьим. Всё это происходит одновременно, потому что три человека работают независимо. Блюда будут готовы раньше, чем если бы их готовил только один повар.

Эти две пары понятий часто путают потому что:

  1. Асинхронный код часто реализуется в однопоточной среде
  2. Многопоточный код часто бывает синхронным, но может быть и асинхронным

К какому виду относится JavaScript?

По умолчанию JavaScript является синхронным и однопоточным языком.

Но как так? Ведь мы нажимаем на кнопки, при этом страницы не зависают, анимации продолжаются, а через секунду появляются результат запросов. И в JavaScript же есть асинхронные операции. Всё дело в Event Loop.

Event Loop или событийный цикл — это механизм, который организует асинхронность в однопоточной среде. Он не делает код многопоточным. Он просто позволяет, во-первых, не блокировать поток на долгих операциях, а во-вторых переключаться между задачами.

Сам по себе событийный цикл невозможен на движке только самого JavaScript. Браузеры, которые являются средами выполнения для JavaScript, предоставляют Web API для создания новых потоков для выполнения JavaScript. Таким образом браузеры дают JavaScript суперспособность: они выносят долгие операции за пределы основного и единственного потока. А как — разберём ниже.

Из чего состоит Event Loop

Event Loop в JavaScript состоит из:

  1. Call Stack: стек вызовов

  2. Task Queue/Macrotask Queue: очередь задач/макрозадач

  3. Microtask Queue: очередь микрозадач

  4. Web API: браузерное окружение

Изначально в модели событийного цикла была только одна очередь задач — Task Queue, куда складывались абсолютно все задачи. Но, с выходом ES6 и появления промисов, была добавлена ещё одна очередь — Microtask Queue, которая нужна, чтобы выполнять короткие, приоритетные задачи сразу после текущего кода, не дожидаясь следующего цикла. Не пугайтесь, что в каких-то моделях может попадаться только одна очередь задач (Task Queue), а в каких-то две — это не является ошибкой.

Для самых дотошных: термин «макрозадача» — это народное название, которое прижилось в сообществе, но его нет в официальной спецификации ECMAScript.

В целом, про Event Loop ничего нет в спецификации JavaScript. Но есть в спецификации HTML, где и описан Event Loop. Там и используется термин «task queue» (очередь задач). В этой спецификации задачи делятся на несколько типов:

  • Tasks — это то, что мы и называем макрозадачами

  • Microtasks — микрозадачи

  • Animation callbacks — задачи рендеринга

Как видно, термин «macrotask» в спецификации не используется. Это придумали разработчики, чтобы отличать их от микрозадач. На практике это не меняет ровно ничего.

Для ещё более дотошных: в спецификации HTML также говорится, что «task queues are sets, not queues» — то есть очереди задач — это не совсем очереди в классическом понимании FIFO. На самом деле это наборы задач, и браузер может выбирать из них задачу с учётом приоритетов, а не строго по порядку поступления. На практике это тоже ничего не меняет.

Принцип работы Event Loop

Event Loop работает очень просто, весь алгоритм можно уместить в 4 шага:

  1. Выполняется ВСЁ, что есть в стеке вызовов и до тех пор пока он не опустеет

    Сюда попадает весь синхронный код при загрузке страницы. После того как синхронный код закончился, он больше не появляется в стеке вызовов в том же виде — до перезагрузки страницы. После этого новые задачи приходят из очередей: макро- и микро-.

    Если бы шаг 1 повторялся постоянно, то браузер бы бесконечно перевыполнял один и тот же код, а макрозадачи и микрозадачи никогда бы не обрабатывались.

  2. Когда стек вызовов станет пустым — выполняются ВСЕ задачи из очереди микрозадач

    Микрозадачи выполняются все подряд, потому что они предназначены для «срочных» операций, которые должны обрабатываться как можно быстрее и без задержек между ними.

    Если бы микрозадачи выполнялись по одной с перерывами на рендеринг, это создало бы задержки между связанными операциями. Например, представим есть цепочка промисов, где каждый вызов создаёт микрозадачу. Если бы эти микрозадачи выполнялись по одной как макрозадачи, а затем после каждой выполнялся бы рендеринг или макрозадача, то порядок бы нарушился. Поэтому их выполняют пачкой — всю очередь микрозадач за раз.

  3. Затем берётся только ОДНУ задача из очереди макрозадач и выполняется

    Сразу после того, как будут выполнены все микрозадачи, но до того, как будет взята следующая макрозадача происходит рендеринг. Браузер использует это окно, чтобы обновить экран, отрисовать анимации и применить изменения в DOM. Если в очереди микро- или макрозадач слишком много, рендеринг может задерживаться — интерфейс будет тормозить, потому что браузеру просто некогда обновлять экран.

    Макрозадачи — это более крупные операции и могут занимать больше времени и ресурсов. Если выполнять их все подряд, браузер не сможет обновлять экран и реагировать на пользователя. Поэтому выполняется только одна макрозадача за раз, а рендеринг происходит до взятия очередной макрозадачи в работу.

    Если стек вызовов опустел, а микрозадач тоже нет, то последовательно могут выполняться все макрозадачи.

  4. Повторять шаги 2-3

    Цикл повторяется, начиная с шага 2: сначала все микрозадачи, потом одна макрозадача — и так до бесконечности. В стек вызовов операции теперь попадают только из очередей микро- и макрозадач.

Далее подробно разберём как работает событийный цикл в JavaScript.

Call Stack

Call Stack или стек вызовов — это место, где выполняется только синхронный код.

Поведение стека вызовов можно описать следующими нехитрыми правилами:

  • Когда вызывается любая синхронная операция она попадает в стек вызовов. Когда заканчивает работу — удаляется из стека вызовов.

  • Функции попадают в стек только при вызове. Объявления функций не попадают в стек вызовов. Возврат из функции — удаляет её из стека.

  • Если внутри функции вызывается другая функция — вложенная кладётся в стеке поверх родительской и сразу начинает выполняться, а родительская функция ждёт, т.е. её выполнение приостанавливается до возврата из вложенной. И так до тех пор, пока стек не опустеет.

  • Функции попадают в стек вызовов в том порядке, в котором они вызываются, а выполняются и удаляются в обратном порядке — с последней вызванной по принципу LIFO т.е. последним пришёл — первым ушел.

    LIFO (Last In, First Out) — это принцип обработки данных, является основой для работы такой структуры структуры данных как стек.

    Пример из жизни: представьте стопку тарелок. Вы можете положить новую тарелку только сверху и взять самую верхнюю тоже первой. В программировании стек вызовов работает так же как стопка тарелок. Основная функция программы вызывает другую функцию, та вызывает третью. Выполнение завершается с самой последней добавленной функции, после чего программа «возвращается» на шаг назад.

  • Пока стек не опустеет, никакие асинхронные задачи из очередей не выполняются.

Пример

Рассмотрим пример с двумя функциями:

1function doSomething() {
2  console.log('Сделать то');

4}
5

Разберём по шагам что в этот момент происходит в Event Loop:

  1. Строки 1-4 и 6-8 — объявления функций, поэтому они в стек вызовов не попадают. На строке 10 вызывается функция, поэтому doSomething() попадает в стек вызовов. Больше в стек на данный момент ничего не поступает, функция doSomething()оказывается на верхушке стека и начинает выполняться.

  2. Выполняется функция doSomething(): вызывается console.log("Сделать то"). console.log() — функция, поэтому она тоже попадает в стек вызовов. В консоли выведется текст «Сделать то», затем console.log("Сделать то") удалится из стека.

    Почему следом в стек не попадает doSomethingElse()? Потому что стек не планировщик и не умеет смотреть в будущее. Он не видит, что после одной функции есть другие.

    Стек работает синхронно:

    1. Берётся текущая операция

    2. Если это вызов функции — функция попадает в стек и выполняется

    3. Когда она завершается — стек очищается

    4. Интерпретатор переходит к следующей инструкции

    Главное правило: обрабатывается ОДНА операция за раз. Что ждёт дальше — пока что неизвестно, важна только текущая операция.

  3. Далее внутри функции doSomething() идёт вызов функции doSomethingElse(). Эта функция тоже отправится в стек вызовов и затем начинается выполняться. Вложенная функция содержит только вывод в консольconsole.log("Сделать сё"), аналогично своему товарищу из предыдущего шага, он тоже попадёт в стек вызовов, выполнится, в консоли выведется текст, а затем console.log("Сделать сё") удалится из стека вызовов.

  4. В стеке вызовов на вершине остаётся функция doSomethingElse(). В ней все операции выполнили, удаляем её из стека вызовов.

  5. В стеке вызовов остаётся функция doSomething(). В ней все операции выполнили, удаляем её из стека вызовов.

  6. Стек вызовов пустой. По алгоритму затем провериться очередь микрозадач — там пусто, и очередь макрозадач — там тоже ничего нет. Пока что делать больше нечего.

Macrotask Queue, Microtask Queue

Мы уже знаем, что в событийной модели есть две очереди задач:

  • Task/Macrotask Queue: очередь задач/макрозадач (раньше, до выхода ES6 и появления промисов, была только одна очередь задач)

  • Microtask Queue: очередь микрозадач

Если стек вызовов работает по принципу LIFO, то очереди задач работают строго по принципу FIFO (First In, First Out) — кто первым пришёл, тот первым и ушёл. Пример из жизни: очередь в магазине — кто раньше встал к кассе, того раньше обслужили.

Почему именно так? Разные принципы — потому что у стека и очереди разное назначение. В стеке важно сохранить вложенность, а очереди важно сохранять порядок поступления задач.

Из названий очередей можно догадаться, что в очередь макрозадач попадают макрозадачи, а в очередь микрозадач — микрозадачи. Так в чём же разница?

Микрозадачи

Это маленькие, но высокоприоритетные задачи. Они создаются для выполнения кода, который должен сработать сразу после завершения текущего синхронного блока, но до того, как браузер или среда выполнит что-либо другое, например, отобразит изменения на экране.

К микрозадачам относятся:

  • промисы и всё, что с ними связано:

    • обработчики промисов .then(...), .catch(...), .finally(...)

      В очередь микрозадач попадают только сами колбэки внутри методов.then(...), .catch(...), .finally(...). Сами эти методы в коде выполняются синхронно, но их переданные функции-аргументы откладываются для асинхронного запуска в очередь микрозадач.

    • async/await — это просто синтаксический сахар для существующего API промисов

    • функция fetch() для отправки сетевых запросов — под капотом имеет тот же промис

      Обратите внимание: часто можно встретить ошибку, в которой функцию fetch() относят к макрозадачам . Это никак не может быть правдой из-за промисной природы функции. Скорее всего путаница появилась из-за того, что fetch() является частью Web API, а где Web API, там и макрозадачи. Поэтому относить fetch() к макрозадачами не просто упрощение, а грубая ошибка потому что:

      • сама по себе функция fetch() — это синхронная операция, т.е. попадёт в стек вызовов, но её результатом будет промис

      • сетевой запрос, созданный fetch(), отправится в Web API (о нём будет ниже), но это не задача в очереди — ни в очереди макрозадач, ни в очереди микрозадач

      • колбэки в .then(...) и .catch(...) — это микрозадачи как у любого промиса

      • сами по себе методы .then(...) и .catch(...) же будут выполняться как синхронные операции и попадут в стек вызовов

      Как видите, очередь макрозадач, тут не используется.

  • queueMicrotask() — специальная встроенная функция JavaScript, которая используется для добавления функции в очередь микрозадач.

    Например, вывод в консоль — это, вообще-то, синхронная операция, но с помощью функции queueMicrotask() мы можем лёгким движением руки сделать её асинхронной:

    snippet(3).js
    1queueMicrotask(() => {
2  console.log('Хоба, я выполнюсь асинхронно');
3})

    Сильно этой функцией злоупотрепблять тоже не надо: есть риск забить очередь микрозадач и тогда ничего кроме микрозадач не будет выполняться.

Алгоритм по шагам будет выглядеть следующим образом:

  2. Доходим до промиса. Promise.resolve() — это синхронная операция, промис создаётся сразу уже выполненными. Поэтому он попадёт в стек вызовов, выполнится, статус промиса перейдёт в успешно выполенно и всё это удалится удалится быстренько из стека вызовов.

  4. В стек вызовов попадает последняя строчка синхронного кода console.log("Конец"). В консоли выводится текст «Конец». Затем функция удаляется из стека вызовов.

  5. Стек вызовов становится пустым. Event Loop видит, что стек вызовов пуст и заглядывает в очередь микрозадач, чтобы выполнить оттуда ВСЕ задачи. Там лежит один единственный колбэк промиса.

  6. Event Loop забирает колбэк промиса и помещает его в стек вызовов. Вот так асинхронные операции окольным путём попадают таки в стек вызовов.

    7. А если промис ещё не будет выполнен на момент, когда стек вызов будет пустой и Event Loop дойдёт до очереди микрозадач? Например:

    snippet(4).js

2  setTimeout(() => resolve(), 100000);

5});

    Если промисов несколько, Event Loop выполнит их все подряд. Если бы в очереди было 1000 промисов — они все выполнились бы строго друг за другом.

А если добавить к промисам setTimeout? Для этого нужно разобраться что же такое макрозадачи и Web API.

Макрозадачи

Это крупные задачи, из которых в основном и состоит работа веб-приложений: таймеры, события DOM-дерева.

Конкретизируем что относится к макрозадачам:

  • Таймеры: setTimeout, setInterval

  • События DOM: scroll, click, focus, mouseup, keydown, submit и т.д. и т.п.

Приоритет выполнения

Обе очереди имеют разный приоритет выполнения — у очереди микрозадач приоритет выше.

Почему именно так? Потому что балом правят промисы. Промисы нужны для реакции на результат асинхронной операции как можно скорее. Если бы они ждали своей очереди как, например, setTimeout, то пришлось бы ждать, пока выполнятся все таймеры. А это долго. Микрозадачи — это срочное сообщение, которое надо прочитать как можно скорее.

Чтобы рассмотреть примеры с очередью макрозадач, забежим вперёд и рассмотрим что такое Web API подробнее.

Web API

Web API — это браузерное окружение, которое находится вне движка JavaScript. Web API выполняет асинхронные операции, пока поток JavaScript занимается другими делами. Без Web API Event Loop был бы просто циклом, который ничего не умеет ждать.

Web API не «склад для колбэков», а диспетчер, который берёт задачу («поставь таймер», «скачай данные», «жди клика»), выполняет её на фоне, а когда заканчивает — отправляет колбэк в очередь.

Web API работает ни по принципу LIFO, ни по принципу FIFO. Это вообще не очередь, а набор независимых таймеров и обработчиков, которые живут своей жизнью.

Что попадает в Web API:

  • Таймеры: setTimeout, setInterval

  • События DOM: scroll, click, focus, mouseup, keydown, submit и т.д. и т.п.

  • Сетевые запросы

Напоминает список из раздела про макрозадачи, верно? Да, но не совсем. Когда появится убеждение, что макрозадачи и Web API связаны, главное вовремя остановить его. Макрозадачи ≠ Web API. После того, как Web API завершил работу (протикал таймеры, получил ответ на запрос, словил клик по кнопке), он отправит в соответствующую очередь колбэки. Этой очередью может быть как очередь макрозадач так и очередь микрозадач, в зависимости от операции, которая выполнялась в Web API:

  • если в Web API закончил тикать таймер, то колбэк отправится в очередь макрозадач

  • если в Web API был получен ответ на сетевой запрос, то колбэк отправится в очередь микрозадач

  • если Web API словил клик по кнопке, то колбэк отправится в очередь макрозадач

Если вы знакомы с JavaScript, вы должны знать, что объект console не является частью самого JavaScript, а является частью Web API. Почему же тогда вызовы console.log() попадают в стек вызовов, а не отправляются в Web API?

Ответ прост: console.log() — синхронная операция. Она выполняется здесь и сейчас, не требует ожидания, не отправляет колбэки в очереди. Вывод в консоль не обращается к сети, не ставит таймеры, не ждёт действий пользователя. Консоль браузера готова принять сообщение в ту же секунду. Web API нужен именно для ожидания: чтобы не блокировать главный поток на время, пока сервер ответит или таймер протикает.

Рассмотрим следующий пример:

  1. Первая строчка синхронного кода console.log("Раз") попадает в стек вызовов, выполняется, в консоли выводится «Раз», затем функция удаляется из стека вызовов.

  2. Доходим до setTimeout. Сам по себе вызов этой функции — синхронная операция, поэтому setTimeout появится в стеке вызовов, выполнится и удалиться. JavaScript понимает, что нужно отправить таймер в Web API. Там таймер будет тикать независимо от основного потока, а JavaScript продолжит дальше идти по коду. Колбэк таймера при этом не выполняется сразу же и пока что никуда не попадает, а ждёт завершения таймера.

  3. Идём дальше по коду, доходим до console.log("Три"), функция попадает в стек вызовов и выполняется, в консоли выводится «Три», затем функция удаляется из стека вызовов.

  4. В Web API всё это время тикал таймер, не блокируя основной поток. Проходит 5 секунд. Браузер замечает, что таймер истёк, и перемещает колбэк таймера в очередь макрозадач.

    Важно: макрозадача появляется уже после того, как Web API закончил работу. Все макрозадачи это колбэки, которые дождались перемещения в очередь макрозадач, где будут ждать своего звёздного часа — когда Event Loop заберёт их в стек вызовов для выполнения.

  5. Стек вызовов пуст. По алгоритму нужно проверить очередь микрозадач и выполнить оттуда ВСЕ микрозадачи, но там ничего нет. Затем по алгоритму нужно проверить очередь макрозадач и взять оттуда на выполнение ОДНУ микрозадачу — там лежит колбэк, который передавали сsetTimeout. Event Loop берёт его из очереди макрозадач и перемещает в стек вызовов.

  6. В стеке вызовов появляется колбэк — console.log('Два'): он выполняется, в консоли выводится «Два» и затем удаляется из стека вызовов.

  3. В Web API вне основого потока выполняется запрос. Основной поток не блокируется, поэтому можно идти дальше по коду: далее выполнится последняя строчка синхронного кода, стек вызовов берёт console.log("Конец"), в консоли появится «Конец», затем функция удалится из стека вызовов.

  4. К этому моменту получаем ответ на сетевой запрос. Вызывается метод then(...) — это тоже просто синхронная операция, поэтому она тоже попадёт в стек вызовов, выполнится и удалится. fetch(), хоть и будет выполняться в Web API, возвращает промис, поэтому его колбэки пойдут в очередь микрозадач. Колбэк console.log('Запрос выполнился') отпрвится в очередь микрозадач.

  5. Стек пуст, Event Loop проверяет очередь микрозадач, берёт оттуда колбэк запроса, перемещает в стек вызовов, выполняет, в консоли появляется «Запрос выполнился», колбэк удаляется из стека вызовов.

Здесь важно понимать, что Event Loop, как только в очереди микрозадач появится колбэк ответа на запрос, не бросит всё, чтобы немедленно пойти выполнять этот колбэк. Например если в этот момент в стеке вызовов выполняется много-много операций (или просто одна какая-то очень долгая), стек вызовов выполнит их все (эту долгую) и только потом пойдёт проверять очередь микрозадач.

Пример

Методы промисов .race(), .all(), .allSettled() и .any() по-разному собирают результаты и по-разному реагируют на завершение промисов. Они сами не создают дополнительных асинхронных операций, а только комбинируют уже запущенные промисы.

Расмотрим гонку запросов на примере метода .race(). Этот метод возвращает промис с результатом первого завершённого из переданных промисов и неважно, оказался ли он успешным или завершился с ошибкой. Остальные же промисы будут продолжаться выполняться в фоне, гонка будет продолжаться, но результаты других промисов будут уже проигнорированы.

Итак, дамы и господа, делаем ставки:

example(8).js
console.log('Делаем ставки');

    fetch('https://www.google.com').then(() => console.log('Гугл ответил')),
    fetch('https://www.yandex.ru').then(() => console.log('Яндекс ответил')),
  ])
  .then(() => {
    console.log('Гонка завершена');
  });

console.log('Ставки сделаны');

Пошагово разберём как этот пример пройдёт через Event Loop:

  1. Первая строчка синхронного кода console.log("Делаем ставки") попадает в стек вызовов, выполняется, в консоли появится «Делаем ставки», затем функция удаляется из стека вызовов.

  2. Доходим до промиса Promise.race(...). Сама по себе эта операция синхронная и попадёт в стек вызовов: выполнится и удалится из стека вызовов. Создаётся промис, внутри начинается перебор массива. Во время этого перебора создадутся два промиса — по количеству запросов, сами запросы уйдут выполняться в Web API (см. следующий шаг). Колбэк Promise.race(...) просто регистрируется и никуда не отправляется. Он попадёт в очередь микрозадач только когда промис Promise.race(...) зарезолвится т.е. как только первый из переданных промисов с fetch() завершится (любым статусом).

  3. Выполнятся два fetch() в следующей последовательности: сперва в стек вызовов попадёт fetch() -- потому что это синхронная операция, затем в Web API появится запрос, fetch() возвращает промис (пока в состоянии pending), затем fetch() удалится из стека вызовов. У каждого запроса есть свой колбэк. Эти колбэки попадут в очередь микрозадач только когда соответствующий запрос завершится. Пока что метод .then() вызывается синхронно на возвращённом промисе чтобы зарегистрировать колбэк.

  4. Выполняется последняя строчка синхронного кода console.log("Ставки сделаны") попадёт в стек вызовов, ввыполнится, в консоли появится «Ставки сделаны», затем функция удалится из стека вызовов.

  5. Стек вызовов пуст, а Event Loop ждёт. Дальше всё зависит от того, какой сервер ответит первым. Допустим, Яндекс ответил быстрее. Web API получает ответ от Яндекса, колбэк этого запроса с console.log('Яндекс ответил') отправляется в очередь микрозадач.

  6. Event Loop видит, что очередь микрозадач не пустая и забирает оттуда задачу: переносит колбэк запроса в стек вызовов, выполняет его, в консоли появится «Яндекс ответил», колбэк удаляется из стека вызовов.

  7. Как только первый промис завершился, а это как раз и произошло, Promise.race(...) переходит в состояние resolved, а его колбэк попадает в очередь микрозадач.

  8. Event Loop забирает следующую микрозадачу: переносит колбэк в стек вызовов, выполняет его, в консоли появится «Гонка завершена», колбэк удаляется из стека вызовов.

  9. Думаете всё? А вот и нет. Web API получает ответ от Гугла, запрос-то никуда не делся. Колбэк с console.log('Гугл ответил')отправляется в очередь микрозадач.

  10. Event Loop забирает последнюю микрозадачу: переносит колбэк в стек вызовов, выполняет его, в консоли появится «Гугл ответил», колбэк удаляется из стека вызовов.

Давайте разберём по шагам:

  1. Первая строчка синхронного кода console.log("Делаем ставки") попадёт в стек вызовов, ввыполнится, в консоли появится «Делаем ставки», затем функция удалится из стека вызовов.

  2. Встречается Promise.all(...). Сама операция попадёт в стек вызовов, выполнится, удалится из стека вызовов. Создадутся два промиса, запросы в каждом уйдут в Web API, при этом первый запрос упадёт, т.к. такой домен не существует, а второй успешно выполнится.

  3. Регистрируются колбэки для каждого запроса, но каждый из них попадёт в очередь микрозадач только когда соответствующий запрос завершится.

  4. Регистрируются колбэки Promise.all(...), их здесь два. Первый колбэк попадёт в очередь микрозадач если все промисы завершатся успешно, а второй если хотя бы один промис завершится ошибкой. Важно помнить, что Promise.all(...) зарезолвится (или упадёт) только когда все промисы завершатся.

  5. Выполняется последняя строчка синхронного кода console.log("Ставки сделаны") попадёт в стек вызовов, ввыполнится, в консоли появится «Ставки сделаны», затем функция удалится из стека вызовов.

  6. Стек вызовов пуст, Event Loop ждёт ответы от серверов. Теперь начинается самое интересное! Сначала приходит ошибка от первого запроса. Web API получает ошибку и промис переходит в состояние rejected. Колбэк же с console.log('Гугл ответил') НЕ вызывается, потому что промис упал.

    Promise.all(...) получает первую ошибку, ему этого достаточно для того чтобы немедленно перейти в состояние rejected (даже если другие промисы ещё не завершились). Колбэк из catch() отправляется в очередь микрозадач.

  7. Event Loop забирает микрозадачу: переносит в стек вызовов, выполняется вывод в консоль «Авария на гонке: TypeError: ...», колбэк удаляется из стека вызовов.

  8. Второй запрос никуда не делся. Через некоторое время приходит успешный ответ от Яндекса. Web API переносит соответствующий колбэк в очередь микрозадач. Event Loop забирает последнюю микрозадачу, переносит в стек вызовов, выполняет, в консоли появляется «Яндекс ответил».

Колбэк c выводом «Гонка завершена» здесь не выполнится, потому что промис Promise.all(...) перешёл в состояние rejected при первой же ошибке, и после этого он уже никогда не перейдёт в fulfilled, даже если потом Яндекс успешно ответит. Этот колбэк выполнился бы только если бы оба промиса завершились успешно. Если нужно дождаться всех промисов, даже с ошибками, можно использовать Promise.allSettled(...).

Макрозадачи, создающие микрозадачи/микрозадачи, создающие макрозадачи/микрозадачи, создаюзие микрозадачи

Да, можно и так, и так, и так. Более того, на практике такое сплошь и рядом:

  • макрозадача может создать микрозадачу: например, внутри setTimeout можно вызвать Promise.resolve().then(...)

  • микрозадача может создать макрозадачу: например, внутри .then(...) можно вызвать setTimeout(...)

  • микрозадача может создать микрозадачу: например, внутри промиса можно создать промис и так до бесконечности:

    snippet(6).js
    1Promise.resolve().then(function loop() {
2  Promise.resolve().then(loop);
3});

    В этом случае стоит быть осторжными: бесконечная генерация микрозадач приведет к зависанию вкладки браузера из-за того, что Event Loop будет бесконечно обрабатывать только очередь микрозадач.

Event Loop не запрещает вкладывать одно в другое. Макрозадачи спокойно создают микрозадачи (промисы внутри таймеров), а микрозадачи — макрозадачи (таймеры внутри промисов). Важно помнить порядок: микрозадачи, накопленные во время выполнения макрозадачи, выполнятся сразу после её завершения, а макрозадачи, созданные внутри микрозадач, попадают в конец общей очереди и ждут следующего цикла. В общем, общий алгоритм работы Event Loop совершенно не меняется.

Начнём с более простого примера: микрозадача (промис) создаёт макрозадачу (таймер):

Promise.resolve().then(() => {
  console.log('Начало микрозадачи');

  setTimeout(() => {

Разберём по шагам что здесь произошло:

  1. Промис попадает в стек, выполняется, колбэк отправляется сразу в очередь микрозадач.

  2. Доходим до console.log('Просто вывод') и как по накатаной: добавляем в стек, выполняем, выводим в консоль «Просто вывод», удаляем из стека.

  3. Код кончился. Стек вызовов пустой. Event Loop идёт в очередь микрозадач, там лежит колбэк промиса. Перемещаем его в стек вызовов, начинаем выполнять. Первой строчкой идёт вывод в консоль: добавляем в стек, выполняем, выводим в консоль «Начало микрозадачи», удаляем из стека. Следующей строчкой идёт setTimeout. Таймер отправляется в Web API (помним, что 0 ms этому не помеха), там оттикивает свои условные 0 ms, затем Web API отправляет колбэк в очередь макрозадач. Продолжаем выполнение колбэка микрозадачи — доходим до последнего вывода в консоль: добавляем в стек, выполняем, выводим в консоль «Конец микрозадачи», удаляем из стека.

  4. Стек вызовов пустой. Очередь микрозадач пустая. Идём в очередь макрозадач, оттуда перемещаем колбэк таймера в стек вызовов и выполняем, в консоли появляется «setTimeout внутри промиса (макрозадача)»

  5. Event Loop возвращается в стек вызовов, но он тоже пустой. В очередях тоже ничего нет. Пока что делать больше нечего.

Таким образом макрозадачи, созданные внутри микрозадач, попадают в конец очереди макрозадач и будут ждать там своей очереди. Всё в рамках алгоритма работы Event Loop. А что будет если всё будет ровно наоборот?

Разберём по шагам что здесь произошло:

  1. setTimeout отправляется в Web API, хоть у него и стоит 0 ms. После этого колбэк с промисом внутри уходит в очередь макрозадач.

  2. Доходим до console.log('Просто вывод'), просто отправляется в стек вызовов, просто сразу выполняется, просто сразу выводится в консоли «Просто вывод» и затем также просто удаляется из стека вызовов.

  3. Идём дальше, доходим до промиса. Сам промис отправляется в стек вызовов, быстренько выполняется и удаляется из стека.

  4. Метод промиса .then(...) просто регистрирует колбэк, который должен будет выполниться после текущего синхронного кода, сам по себе никуда не попадает — ни в стек, ни в очереди задач. Колбэк же помещается в очередь микрозадач.

  5. Код кончился. Что имеем на данный момент: в очереди микрозадач один колбэк от наружнего промиса, в очереди макрозадач один колбэк с промисом внутри setTimeout.

  6. Стек вызовов пуст, Event Loop первым делом идёт в очередь микрозадач и выполняет там всё, что там лежит. У нас там колбэк от наружнего промиса. Колбэк перемещается в стек вызовов, выполняется, в консоли появляется «Промис выполнился (микрозадача снаружи)».

  7. В очереди микрозадач всё выполнили, идём в очередь макрозадач. Там один колбэк с промисом внутри. Перемещаем колбэк в стек вызовов и начинаем выполнять. Первая строчка в колбэке — синхронный вывод. Добавляем в стек, выводим в консоли «Начало макрозадачи», удаляем из стека. Далее доходим до промиса. Сам промис попадает в стек вызовов, быстренько выполняется и сразу удаляется. Метод промиса .then() отправляет колбэк в очередь микрозадач. Идём по коду дальше — снова синхронный вывод. Добавляем в стек, выводим в консоли «Конец макрозадачи», удаляем из стека.

  8. Стек вызовов снова опустел. Идём по порядку: заглядываем в очередь микрозадач, а там нас уже ждёт колбэк промиса, созданного внутри таймера. Перемещаем его в стек вызовов, выполняем, выводим в консоли «Промис выполнился (микрозадача внутри макрозадачи)», удаляем из стека.

  9. Возвращаемся в очередь микрозадач потому что нужно выполнить оттуда все задачи, там больше ничего нет. Заглядываем в очередь макрозадач — тоже пусто. Event Loop возвращается в стек вызовов, но он тоже пустой. Пока что делать больше нечего.

Что из этого важно вынести: Event Loop не прерывает текущую макрозадачу ради микрозадачи — микрозадачи накопятся и выполнятся после завершения макрозадачи, хоть у микрозадач и есть приоритет над макрозадачами. При этом если в очереди макрозадач есть колбэки, которые ожидают своей очереди, а в процессе выполнения одной из макрозадач появились микрозадачи, выполнение оставшихся макрозадач отложится до тех пор, пока все микрозадачи не будут выполнены.

Пример

Повышаем уровень сложности: рассмотрим пример, в котором микрозадачи (промисы), которые создают другие микрозадачи (промисы).

Осторожно: бесконечные микрозадачи! Если внутри микрозадачи создавать новую микрозадачу, Event Loop может застрять в бесконечном цикле. В этом случае браузер никогда не дойдёт до фазы рендеринга и интерфейс зависнет. Это классическая ошибка при работе с промисами и/или queueMicrotask().

DO_NOT_DO_THIS(3).js
1let counter = 0;
2
3function infiniteMicrotaskLoop() {
4  // Добавляем микрозадачу, которая добавляет новую микрозадачу
5  Promise.resolve().then(() => {
6    counter++;
7    // Обновляем текст на странице (но это никогда не отрендерится)
8    document.getElementById('counter').textContent = counter;
9
10    // Рекурсивно добавляем следующую микрозадачу
11    infiniteMicrotaskLoop();
12  });
13}
14
15infiniteMicrotaskLoop();

Чтобы избежать этого можно:

  • использовать макрозадачи для разрыва цепочки микрозадач. Отлично подойдут таймеры с нулевой задержкой setTimeout(callback, 0):

    DO_THIS(1).js
    1function safeLoop() {
2  if (!isRunning) return;
3
4  // Работаем в синхронном коде
5  counter++;
6  document.getElementById('counter').textContent = counter;
7
8  // Разрываем цепочку через макрозадачу
9  setTimeout(safeLoop, 0);
10}

  • использовать requestAnimationFrame для разрыва цепочки, чтобы дать браузеру возможность обновить экран:

    DO_THIS(2).js
    1function animationLoop() {
2  if (!isRunning) return;
3
4  counter++;
5  document.getElementById('counter').textContent = counter;
6
7  // Следующий кадр анимации (до рендеринга)
8  requestAnimationFrame(animationLoop);
9}

  • Следить за количеством микрозадач и не допускать застревания Event Loop в очереди микрозадач.

Как перестать бояться Event Loop

Вы прошли долгий путь. Давайте соберём самое важное, что нужно запомнить по Event Loop в JavaScript:

  1. Call Stack: стек вызовов

    • сюда весь синхронный код
    • выполняется один раз при загрузке страницы, а потом сюда попадают только колбэки
    • LIFO

  2. Microtask Queue: очередь микрозадач

    • сюда промисы
    • выполняется вся целиком после стека, но до макрозадач
    • после всех микрозадач рендерится страница
    • FIFO

  3. Macrotask Queue: очередь макрозадач

    • сюда setTimeout, setInterval, события DOM
    • выполняется по одной задаче за итерацию Event Loop
    • до начала выполнения очередной макрозадачи рендерится страница
    • FIFO

  4. Web API: браузерное окружение

    • выполняются фоново таймеры, слушатели событий, запросы

    • только для асинхронных операций

    • не стек и не очередь

Принцип работы Event Loop можно сократить следующим образом до — выполняем:

  1. ВСЁ в стеке вызовов пока он не опустеет

  2. ВСЕ микрозадачи

  3. ОДНУ макрозадачу

Затем вернуться к шагу 2 и повторять пункты 2-3 до бесконечности.
