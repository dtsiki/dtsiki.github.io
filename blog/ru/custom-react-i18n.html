DIY: локализуем приложение на React сами

13 march 2026

В этой статье изобретём велосипед и создадим своими руками i18next в приложении на React. Контекст, хук и никакой магии.

Содержание

Для локализации приложений обычно используются специальные библиотеки. Например для React часто используется одна из популярных библиотек react-i18next на базе движка интернационализации i18next. В этом же пошаговом уроке вместе напишем свой собственный i18n для React: разберём архитектуру, создадим контекст, сделаем переключение языков и удобный хук для переводов. Подойдёт для небольших проектов и для тех, кто хочет понять, как это работает изнутри.

Планируем архитектуру

Как говорится «Без ТЗ результат ХЗ». Прежде чем писать код, представим, как будущая система локализации будет работать.

Что нам нужно:

  • иметь список языков, на которые можно будет переводить;

  • где-то глобально хранить выбранный язык;

  • функция переключения языка;

  • где-то хранить переводы в каком-то удобном виде;

  • функция использования этих переводов;

  • выбранный язык и функцию переключения языка надо иметь возможность использовать в любом компоненте.

Выглядит страшно, но спойлер: это всё можно сделать буквально в несколько файлов.

Для решения поставленной задачи отлично подойдёт React Context: создадим контекст, в котором будем хранить язык и функцию для переключения языка, а затем сможем получать к ним доступ из любого компонента в приложении с помощью провайдера.

Схематично будущая структура проекта с кастомной локализацией будет выглядеть примерно:

src/
├── main.tsx                          # точка входа
├── App.tsx                           # корневой компонент <App />
├── index.css                         # глобальные стили
│
├── context/                          # все контексты
│   └── LanguageContext/
│      ├── LanguageContext.types.ts   # типы для контекста
│      ├── LanguageContext.tsx        # сам контекст
│      ├── LanguageProvider.types.ts  # типы для провайдера
│      └── LanguageProvider.tsx       # провайдер
│
├── i18n/                             # всё для локализации
│   ├── language.ts                   # Language (константа + тип)
│   ├── translations/                 # переводы по категориям
│   │   ├── common.ts
│   │   └── index.ts
│   ├── translate.ts                  # функция перевода
│   └── index.ts                      # единый экспорт
│

Проект в примере собран с помощью Vite

Ваш итоговый вариант, конечно же, будет зависеть от ваших исходных данных и структуры вашего проекта. Главное тут понять принцип.

Определяемся с языками

Сначала нам нужно решить какие языки будем использовать для локализации в проекте. Пойдём простым путём и пусть будут всего лишь два языка — русский и английский. Раньше для перечисления языков часто использовали перечисления enum:

1export enum ELang {
2  ENG = 'ENG',
3  RU = 'RU',
4}

Однако в современных настройках TypeScript (особенно с isolatedModules: true) использование enum может вызывать ошибки. Поэтому будем использовать более современный подход: объект с as const и выводимый из него тип type. Это даст ту же строгую типизацию, но лучше совместимо со сборщиками нового поколения, например Vite. Будет выглядеть чуть длиннее:

src/i18n/language.ts
1export const Language = {
2  ENG: "English",
3  RU: "Russian",
4} as const;
5
6export type Language = (typeof Language)[keyof typeof Language];

Всё это надо, чтобы не заморачиваться с тем, как правильно потом прописывать названия языков, например может возникнуть путаница сENG, EN илиENGLISH. Со строгим типом будем просто вызывать константу и выбирать нужный язык. Если захотим потом расширить этот список — просто добавим новые языки. Например, если решим добавить в приложение тарабарский язык:

src/i18n/language.ts
1export const Language = {
2  ENG: "English",
3  RU: "Russian",
4  GIB: "Gibberish"
5} as const;
6
7export type Language = (typeof Language)[keyof typeof Language];

Создаём контекст

Создадим интерфейс для контекста ILanguagesContextData, который будет описывать, что будет лежать в контексте. В нём будет всего лишь два свойства — язык и функция переключения языка:

src/LanguageContext/LanguageContext.types.ts
1import type { Language } from "../../i18n/language";
2
3export interface ILanguageContextData {
4  language: Language;
5  toggleLanguage: () => void;
6}

Так как в примере всего два языка, то можно сделать простой переключатель по типу переключателя света - каждое нажатие на кнопку будет менять язык на второй возможный. Если вы хотите более сложный интерфейс или большее количество языков, то просто измените функцию переключения языка, добавив параметр - язык, на который нужно перевести всё, а кнопку замените, например на выпадающий список.

Контекст создаём с начальным значением undefined, потом провайдер всё равно перезапишет его:

src/LanguageContext/LanguageContext.tsx
1import { createContext } from "react";
2import { type ILanguageContextData } from "./LanguageContext.types";
3
4export const LanguageContext = createContext<ILanguageContextData | undefined>(
5  undefined
6);

Контекст готов!

Создаём провайдер

Провайдер будет хранить состояние языка и переключать его для нас.

У провайдера будет только одно свойство - children:

src/LanguageContext/LanguageProvider.types.ts
1import type { ReactNode } from "react";
2
3export interface ILanguageProviderProps {
4  children: ReactNode;
5}

Сам провайдер будет очень простым:

src/LanguageContext/LanguageProvider.tsx
1import { useState } from "react";
2import type { ILanguageProviderProps } from "./LanguageProvider.types";
3import { LanguageContext } from "./LanguageContext";
4import { Language } from "../../i18n/language";
5
6export const LanguageProvider = ({ children }: ILanguageProviderProps) => {
7  const [language, setLanguage] = useState<Language>(Language.ENG);
8
9  const toggleLanguage = () => {
10    setLanguage((prev) => (prev === Language.ENG ? Language.RU : Language.ENG));
11  };
12
13  return (
14    <LanguageContext.Provider value={{ language, toggleLanguage }}>
15      {children}
16    </LanguageContext.Provider>
17  );
18};

Что происходит в провайдере:

  • С помощью хука useState храним текущий язык, по умолчанию это русский
  • Функция toggleLanguage просто переключает язык между двумя возможными
  • Провайдер отдаёт значение всему, что внутри него с помощью свойства children

Далее нужно обернуть корневой компонент этим провайдером:

src/main.tsx
1import { StrictMode } from "react";
2import { createRoot } from "react-dom/client";
3import App from "./App.tsx";
4import { LanguageProvider } from "./context/LanguageContext/LanguageProvider.tsx";
5
6import "./index.css";
7
8createRoot(document.getElementById("root")!).render(
9  <StrictMode>
10    <LanguageProvider>
11      <App />
12    </LanguageProvider>
13  </StrictMode>
14);

Создаём хук

Хранить язык в контексте — это хорошо, но каждый раз писать useContext(LanguageContext) и проверять на undefined не очень удобно:

HelloWorld.tsx
1import { useContext } from 'react';
2import { LanguageContext } from '../context/LanguageContext/LanguageContext';
3import { HELLO_WORLD } from '../i18n';
4
5export const HelloWorld = () => {
6  // Каждый раз импортируем useContext и LanguageContext и каждый раз проверяем, что контекст не undefined
7  const context = useContext(LanguageContext);
8
9  // Эта проверка будет в КАЖДОМ компоненте
10  if (!context) {
11    throw new Error('Компонент должен быть обёрнут в LanguageProvider');
12  }
13
14  // Каждый раз достаём язык из контекста
15  const { language } = context;
16
17  // Пишем функцию перевода прямо в компоненте (или импортируем отдельно)
18  const translate = (translations: TranslationObject) => {
19    return translations[language];
20  };
21
22  return (
23    <p>{translate(HELLO_WORLD)}</p>
24  );
25};

Так делать не будем: в этом примере всё выглядит невинно, но чем больше компонент и чем больше будет самих компонентов, то тем тяжелее будет таскать и поддерживать за собой кучу ненужного кода. Помните: DRY!

Сделаем специальный хук useTranslate, который будет инкапсулировать (ого, мы прям настоящие программисты!) всю логику в одном месте и нам не нужно будет таскать во всех компоненты за собой одинаковый код.

src/hooks/useTranslate.tsx
1import { useContext } from "react";
2import { LanguageContext } from "../context/LanguageContext/LanguageContext";
3import type { TranslationObject } from "../types/i18n.types";
4
5export const useTranslate = () => {
6  const context = useContext(LanguageContext);
7
8  if (!context) {
9    throw new Error(
10      "useTranslate должен использоваться внутри LanguageProvider"
11    );
12  }
13
14  const translate = (translations: TranslationObject): string => {
15    return translations[context.language];
16  };
17
18  return {
19    translate,
20    language: context.language,
21    toggleLanguage: context.toggleLanguage,
22  };
23};

А будем делать так: перенесём этот код в хук и будем вызывать этот хук там, где нам нужно что-то перевести

Что тут происходит:

  1. Получаем контекст с помощью useContext
  2. Проверяем, что контекст существует (защита от глупых ошибок)
  3. Создаём функцию-переводчик translate, которая просто берёт нужное поле из объекта.
  4. Возвращаем наружу функцию-переводчик translate, функцию переключения toggleLang и язык

Организуем хранение переводов

Мы подошли к вопросу где и в каком виде хранить переводы. Для этого создадим папку i18n в директории проекта.

Для переводов нам потребуется специальный объект:

src/types/i18n.types.ts
1import type { Language } from "../i18n/language";
2
3export interface TranslationObject {
4  [Language.ENG]: string;
5  [Language.RU]: string;
6}

Этот объект поможет нам типизировать объекты перевода. Он будет говорить TypeScript'у: «Любой объект, который хочет быть переводом, должен содержать поля English и Russian, и эти оба поля должны быть строками». Таким образом, когда будем составлять словари, избежим дурацких ошибок:

src/i18n/translations/common.ts
1// ❌ Ошибка: нет перевода на русский
2const HELLO_WORLD: TranslationObject = {
3  [Language.ENG]: "Hello, world!",
4};
5
6// ❌ Ошибка: значение не строка
7const HELLO_WORLD: TranslationObject = {
8  [Language.ENG]: 42,
9  [Language.RU]: "Привет, мир!",
10};

Пример дурацких ошибок при составлении объектов, благодаря TranslationObject эти ошибки будут подсвечены сразу же

Далее создадим дочернюю папку translations и будет складывать туда файлы c объектами с переводами в виде:

src/i18n/translations/common.ts
1import type { TranslationObject } from "../../types/i18n.types";
2import { Language } from "../language";
3
4export const HELLO_WORLD: TranslationObject = {
5  [Language.ENG]: "Hello, world!",
6  [Language.RU]: "Привет, мир!",
7};

Названия объектов написаны капсом чтобы в компонентах было сразу понятно: «ага, это же перевод!».

Внутри папки i18n переводы можно разделять по смыслу по файлами, например в файл common.ts складывать переводы общих компонентов интерфейса, а в папку blog.ts переводы для блога и т.д. и т.п.

Переводим

Осталось дело за малым - применить всё на деле.

Мы уже обернули корневой компонент приложения провайдером, поэтому чтобы использовать переводы достаточно вызвать в нужном компоненте хук useTranslate и использовать из него функцию перевода translate. В неё нужно будет просто передать нужный объект с переводом из папки i18n. Например:

src/components/HelloWorld.tsx
1import { useTranslate } from "../../../hooks/useTranslate";
2import { HELLO_WORLD } from "../../../i18n";
3
4export const HelloWorld = () => {
5  const { translate } = useTranslate();
6
7  return <p>{translate(HELLO_WORLD)}</p>;
8};

Добавляем переключатель языка

У нас в примере всего два языка, поэтому кнопку смены языка можно сделать в виде переключателя - просто нажимаем на неё и язык меняется на второй возможный:

src/components/LanguageSwitcher.tsx
1import { useTranslate } from "../../../hooks/useTranslate";
2import { Language } from "../../../i18n";
3
4export const LanguageSwitcher = () => {
5  const { toggleLanguage, language } = useTranslate();
6
7  return (
8    <button className="button" onClick={toggleLanguage}>
9      {language === Language.ENG ? Language.RU : Language.ENG}
10    </button>
11  );
12};

Этот компонент можно использовать в меню, заголовках, футере и в любых других компонентах приложения вне зависимости от вложенности — переключение языка будет работать везде.
