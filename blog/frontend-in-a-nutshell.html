JavaScript

JavaScript is a scripting language that allows to implement different features on web pages: dynamically updating content, control, animate staff, etc, etc, etc.

First at all, JavaScript is a dynamic language with dynamic types. Variables are not directly associated with any type and can be assigned or reassigned at any moment. In the example below a variable yay was initialized with one type, then it was reassigned twice by modifying the value. What are these types?

index.js
1let yay = 42;
2yay = 'yay';
3yay = true;

There are only two kind of data types in JavaScript: primitive and reference data types. Primitive types are pretty simple as the name suggests. They are so simple that they don't have methods. Wait, stop - you might say, strings actually have methods! In fact, JavaScript converts primitive types to objects (it's called Object wrapper), so this is why it is possible to use string object methods.

typeof operator

We can use typeof operator to find the data type of JavaScript variables. If we use this operator along with each of primitive type, it will return a type of the same name:

index.js
1console.log(typeof 'yay'); // "string"
2console.log(typeof 42); // "number"
3console.log(typeof true); // "boolean"

All these types, except null and undefined, have their corresponding object wrapper types or class, for example: number - Number, string - String, etc. As a result, these object wrappers provide us methods for primitive types, for exampe toUpperCase() for numbers or toLowerCase() for strings.

And, last but not least, objects are JavaScript data types. While primitive types can contain only a single thing, objects are used to store collections of key-value pairs.typeof object command always returns object.

Thus, almost everything in JavaScript is an object. In fact, only six things are not objects: null , undefined , strings, numbers, boolean, and symbols. Object wrappers, thanks for that!

A mutable object is an object whose state can be modified after it is created. An immutable object is an object whose state cannot be changed once the object is created.

In JavaScript only objects and arrays are mutable, primitive data types are immutable. Let's break it down.

There are two data types in JavaScript: primitive types (string, number, boolean, etc) and non-primitive or reference data type (object). Primitive types are stored on the stack memory. When storing a Primitive type in memory, it adds an element to the top of the stack with this value of the newly created variable, in other words last in, first out (LIFO). When we create a new variable and assigned the first one to the new one, it adds a new element on top of the stack with the value of the new variable. Reference types are stored on the heap. It, indifference from the stack, has no order of how to store the data. When storing a reference type in memory, it adds a new element to the top of the stack when its value is a reference to the address of the object that has been stored on the heap.

Each time a primitive type is created, a new element gets into the top of the stack and stores the data of that variable. On the other hand, when create a reference data type, a new element gets into the top of the stack, but this time it stores its poitner to the address location of the object in the heap. Then, when we assign the created object to a new object variable a new element gets into the top of the stack but with the same pointer to the first object.

JavaScript provides functions similar to most of the languages. A function is a block of code designed to perform a particular task. All the functions in JavaScript can use arguments object by default or may not have any arguments.

To create a function in JavaScript we can use different approaches:

  1. function declaration

    A function created with a function declaration is a Function object and has all its properties, methods and behavior of Function objects.

    index.js
    1function doSomething() {
2  console.log('yay');
3}
4
5doSomething();

    Functions declared with function declaration return undefined value by default. To return any other value, the function must have a return statement:

    index.js
    1function getAnswerToLifeTheUniverseAndEverything() {
2  return 42;
3}
4
5console.log(getAnswerToLifeTheUniverseAndEverything());
    > 42

    Furthermore, function declarations in JavaScript are hoisted to the top of the enclosing function or global scope:

    index.js
    1doSomething();
2
3function doSomething() {
4  console.log('yay');
5  console.log('qeq');
6}

    In the terminal you will find the following output after executing this code snippet. As you can see, any statements that operate on a function still run in normal order.

    > "yay"
> "qeq"
  2. function expression

    A function expression is very similar to a function declaration. Here's the code snippet of an unnamed function declaration:

    index.js
    1const doSomething = function() {
2  console.log('yay');
3};
4
5doSomething();
    > "yay"

    The main difference between function expressions and function declarations is the function name: JavaScript allows to assign a function to a variable and then use that variable as a function:

    index.js
    1const calculate = function getAnswerToLifeTheUniverseAndEverything() {
2  return 42;
3};
4
5const result = calculate(); // 42
6getAnswerToLifeTheUniverseAndEverything(); // ReferenceError: getAnswerToLifeTheUniverseAndEverything is not defined

    Function expressions, unlike function declarations, are not hoisted:

    index.js
    1doSomething();
2
3var doSomething = function () {
4  console.log('yay');
5};
    > TypeError: doSomething is not a function

    A function expression can be used as an IIFE or Immediately Invoked Function Expression which runs as soon as it is defined.

    index.js
    1(function () {
2  console.log('Yay!');
3})();
  3. arrow function expression

    An arrow function expression is a alternative to a regular function expression:

    index.js
    1const doSomething = () => {
2  console.log('yay!');
3}

    Arrow functions can have a concise body:

    index.js
    1const doSomething = () => console.log('yay!');

    Arrow functions don't have their own bindings to this, arguments, or super. Also they should not be used as methods. Arrow functions cannot be used as constructors too. Calling them with new throws a TypeError.

Scope means where variables are available for use. There are three types of scope in JavaScript:

  1. Global scope
  2. Block scope
  3. Function scope

Mentioned above var declarations are globally scoped. This means that any variable that is declared with var outside a block is available for use in the whole window. Whoa!

index.js
1if (true) {
2  var yay = 'yay';
3}
4
5console.log(yay); // "yay"

This occurs because var declarations, wherever they occur, are processed before any code is executed. This proccess is called hoisting and we'll explore it in detail later. This is how the example above looks like in reality in JavaScript:

index.js
1var yay;
2
3if (true) {
4  yay = 'yay';
5}
6
7console.log(yay); // "yay"

There is a trick called function scope. When a variable is declared inside a function, it is only accessible within that function and cannot be used outside that function. The example above would look like this:

index.js
1var yay = 'yay';
2
3function doSomething() {
4  var qeq = 'qeq';
5}
6
7console.log(yay); // "yay"
8console.log(qeq); // ReferenceError: qeq is not defined

Variable yay, even though it is declared using var, is globally scoped because it exists outside a function while qeq is function scoped. We cannot access any variable outside of a function, neither const or let:

index.js
1const yay = 'yay';
2
3function doSomething() {
4  const qeq = 'qeq';
5}

7console.log(yay); // "yay"
8console.log(qeq); // ReferenceError: qeq is not defined

1let yay = 'yay';

3function doSomething() {
4  let qeq = 'qeq';
5}

Below is an example of classical hoisting. Here you can see calling a function before defining it.

index.js
1doSomething();

The reason is hoisting. Hoisting is a JavaScript mechanism where variables and function declarations are moved to the top of their scope before code execution. Javascript statements are executed in the order they appear. In other words, JavaScript executes code from top to bottom. That is until hoisting get involved into game. Function declaration statements are always hoisted to their containing scope, so their functions exist even before the statements appear. But any statements that operate on a function still run in normal order. Unfortunately, it doesn't apply to anonymous functions or arrow functions because we cannot call them before initialization:

index.js

2
3const doSomething = () => {
4  console.log('yay');
5}

In the terminal you will find the following output after executing the code snippet above. We will get exactly the same result for arrow function with let keyword.

> ReferenceError: can't access lexical declaration 'doSomething' before initialization

But there is trick for var arrow function:

index.js

2
3var doSomething = () => {
4  console.log('yay');
5}
> TypeError: doSomething is not a function

Why that happened? Variable declarations using var keyword are also hoisted, are not? This occurs because variable assignments actually are not hoisted. The var doSomething variable will contain undefined until the line of code that does the assignment is executed. In other words, the code snipper below is equal the code snippet above:

index.js

2
3var doSomething = undefined;
> TypeError: doSomething is not a function

A temporal dead zone (also known as TDZ) is the area of a block where a variable is inaccessible until the moment the computer completely initializes it with a value. A block is a pair of curly braces. A block's temporal dead zone starts at the beginning of the block's local scope. It ends when variable is fully initialized with a value. Here's the example of TDZ of variables declared with let keyword:

index.js
1{// TDZ starts here
2  // TDZ continues here
3  console.log(yay); // ReferenceError: can't access lexical declaration 'yay' before initialization
4  // TDZ continues here
5  let yay = "yay"; // TDZ ends here
6  // TDZ does not exist here
7}

In the example above TDZ of the yay variable starts from the opening curly bracket and ends once when it initialized with the value. The yay variable was invoked within its temporal dead zone that is why JavaScript will return a ReferenceError.

Here is the example how we can access a variable successfully after its complete initialization:

index.js
1{// TDZ starts here
2  // TDZ continues here
3  // TDZ continues here
4  let yay = "yay"; // TDZ ends here
5  // TDZ does not exist here
6  console.log(yay); // "yay"
7  // TDZ does not exist here
8}

Another difference between var, let and const variables is when their TDZ ends. Let's look at an example of TDZ of var:

index.js
1{// TDZ starts and ends here
2  // TDZ does not exist here
3  console.log(yay); // undefined
4  // TDZ does not exist here
5  var yay = "yay"; // TDZ does not exist here
6  // TDZ does not exist here
7  console.log(yay); // "yay"
8  // TDZ does not exist here
9}

In contrast to let or const, JavaScript automatically initializes var variables with undefined value.let and const both remains dead and inaccessible. However, var TDZ ends immediately after its hoisting.

Strict mode was introduced in ES5. This mode prevents certain actions from being taken and throws more exceptions. Strict mode makes several changes to normal JavaScript semantics.

Strict mode can be used in two ways:

  1. For the entire script in global scope
  2. For certain functions

Along with this, strict mode doesn`'t work with block statements enclosed in curly braces.

To invoke strict mode for an entire script just put the following statement before any other statements:

index.js
1"use strict";
2...
3...
4...

To invoke strict mode for a function put the exact statement in the body of function before any other statements:

index.js
1function doSomething() {
2  "use strict";
3  ...
4  ...
5  ...
6}

Pay attention: the entire contents of JavaScript modules are automatically in strict mode. Also all parts of a class's body are strict mode code too.

Synchronous programming is a programming model where operations take place sequentially step by step:

index.js
1const yay = 'yay';
2const qeq = 'qeq';
3const yayQeq = `${yay} ${qeq}`;

This code above declares a string called yay, then declares a string called qeq and then declares another string called yayQeq, which uses yay and qeq. In other words, the browser steps through this code one line at a time in the order it was written.

What if the synchronous function takes a long time? If any function will execute too long or will be long-running, the whole program can be completely unresponsive: you cannot type anything, click anything or do anything else. There are asynchronous functions to address this issue.

With asynchronous programming, JavaScript programs can start long-running tasks and continue running other tasks in paralell.

By default, JavaScript is a synchronous and single threaded programming language. What are we supposed to do, if we need to fetch some large amount of data from a backend and then display it on our web page? With the help of callbacks, async/await and promises, JavaScript is used to do asynchronous programming.

A callback is a function that is passed inside another function, and then called in that function to perform a task. You've already faced with callbacks.

index.js
1setTimeout(() => {
2  console.log('yay');
3}, 1000);

The setTimeout() is a JavaScript function that takes two parameters: an another function and time in milliseconds after which that function should be executed. So console.log('yay') is actually the callback in the example above.

Using callbacks was very efficient, but only to a certain point (e.g. using synchronous callbacks is still normal practice) and now using asynchronous callbacks is a bad practice. If synchronous callbacks are ways to achieve greater programming flexibility, then asynchronous callbacks are ways to achieve greater performance and user experience. The phenomenon which happens when we nest multiple callbacks within a function is called a callback hell.

GALLERYcallback-hell.jpg

To fix this problem promises were introduced.

Promises came along to solve the problems of callback functions. Promise is a special JavaScript object which contains its state and produces a value after some asynchronous operation completes successfully or unsuccessfully. This promise's state can be:

  • pending
    while the state is pending the result is undefined
  • fulfilled
    when the state is fulfilled the result is a value
  • rejected
    when the state is rejected the result is an error object

Initially state is pending, then changes to either fulfilled when resolve is called or rejected when reject is called.

The constructor syntax for a promise object is:

index.js
1const promise = new Promise(function(resolve, reject) {
2  // executor
3});

The function passed to new promise is called the executor. The executor performs a job, usually it is something that takes time.

Its arguments resolve and reject are callbacks provided by JavaScript itself. When the executor obtains the result, it should call one of these callbacks. Callback resolve(value) will call if the job is finished successfully with result value:

index.js
1const promise = new Promise(function(resolve, reject) {
2  resolve('I am done');
3});
4
5promise.then((result) => {
6  console.log(result);
7});
> "I am done"

Callback reject(error) will call if an error has occurred, where error is the error object:

index.js
1const promise = new Promise(function(resolve, reject) {
2  reject(new Error('Something went wrong'));
3});
4
5promise
6  .then((result) => {
7    console.log(result);
8  })
9  .catch((error) => {
10    console.error(error);
11  });
> [Object error]

There can be only a single result: a value or an error. Here's the example:

index.js
1let promise = new Promise(function(resolve, reject) {
2  resolve('yay');
3
4  reject(new Error('Error')); // ignored
5
6  setTimeout(() => resolve('qeq')); // ignored
7});

Promises can be chained as we touched above. JavaScript provide a couple of methods to do that: then, catch and finally. All these methods, in its turn, return promises. The chain is composed of then calls, and typically but not necessarily has a single catch at the end and optionally followed by finally. Promises chaining looks like this:

index.js
1new Promise(function(resolve, reject) {
2
3  setTimeout(() => resolve(1), 1000);
4
5}).then(function(result) {
6
7  console.log(result);
8
9  return result * 2;
10
11}).then(function(result) {
12
13  console.log(result);
14
15  return result * 2;
16
17}).then(function(result) {
18
19  console.log(result);
20
21  return result * 2;
22}).finally(() => {
23  console.log("All done")
24)};
> 1
> 2
> 4
> "I am done"

Apart from the handler methods then, catch and finally, there are six static methods for handle multiple promises.

How can promises be used? In frontend, promises are often used for network requests.

How to handle asynchronous code in JavaScript? One way is to use the async and await keywords which allow to write asynchronous code in a synchronous-looking style.

To define an async function just add async keyword:

index.js
1async function doSomethingAsync() {
2  // do something async here
3}

Async functions may also be defined as expressions:

index.js
1const doSomethingAsync = async() => {
2  // do something async here
3};

Async functions can contain zero or more await expressions. The await keyword make functions behave as though they're synchronous by suspending execution until the returned promise is fulfilled or rejected. That's why is calling an async function will always return a Promise.

Keep in mind: the await keyword is only valid inside async functions. If use it outside of an async function, we will get a SyntaxError.
